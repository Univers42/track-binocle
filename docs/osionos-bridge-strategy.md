# osionos Bridge Strategy

## Goal

opposite-osiris remains the public website and authentication entry point. osionos remains the private application workspace. Both use the same mini-BaaS stack, but they do not duplicate backend services and neither browser app receives database secrets, service-role keys, SMTP credentials, or raw profile records from the other app.

## Chosen Architecture

The bridge is a server-to-server handoff with signed, short-lived app sessions.

1. The user signs in on opposite-osiris through the shared BaaS auth gateway.
2. opposite-osiris calls `POST /api/auth/osionos-session` with the user's BaaS access token.
3. The opposite-osiris gateway verifies the token with BaaS `/auth/v1/user`.
4. The gateway creates a minimal bridge payload: provider, user id, display name, email, and a random replay nonce.
5. The gateway signs that payload with `OSIONOS_BRIDGE_SHARED_SECRET` using HMAC SHA-256 and sends it to osionos.
6. The osionos bridge verifies timestamp skew, HMAC signature, payload shape, and replay nonce before doing anything else.
7. The osionos bridge hashes the email and upserts an owner workspace through the shared BaaS RPC `osionos_bridge_upsert_workspace`.
8. osionos returns a one-time `bridge_token` redirect URL to the browser.
9. The osionos frontend consumes that token through `POST /api/auth/bridge/consume` and stores only the app-scoped session/workspace model.

## Security Boundaries

Only server processes know these secrets:

- `OSIONOS_BRIDGE_SHARED_SECRET`
- `OSIONOS_APP_SESSION_SECRET`
- `BAAS_SERVICE_ROLE_KEY`
- `JWT_SECRET`

The browser receives only:

- a short-lived one-time bridge token in the osionos URL
- an app-scoped osionos access token generated by the osionos bridge
- the private workspace id/name/slug and owner permissions needed by the app UI

The bridge does not forward database connection strings or service-role keys to either frontend. Email is used only inside the trusted bridge process and is stored in the osionos mapping as a SHA-256 hash, not as a raw address.

## ABAC Model

The BaaS migration [models/osionos-bridge-migration.sql](../models/osionos-bridge-migration.sql) creates:

- `osionos_bridge_identities`: one Prismatica subject mapped to one osionos user/workspace.
- `osionos_workspaces`: the user's private osionos workspace.
- `osionos_workspace_members`: owner membership with `create`, `read`, `update`, `delete`, and `admin` permissions.
- `osionos_bridge_audit_events`: audit trail for bridge upserts.

When the shared ABAC tables exist, the RPC also creates an `osionos_owner` role and a workspace-scoped `resource_policies` entry. This gives the authenticated user the highest permission level on their own osionos workspace without granting access to other users' workspaces.

## Runtime Services

The root Compose stack now includes one osionos bridge API service instead of duplicating BaaS services:

- `kong`, `gotrue`, `postgrest`, `postgres`, and related services remain shared.
- `auth-gateway` runs the opposite-osiris auth entry point.
- `osionos-bridge` runs [apps/osionos/app/scripts/bridge-api.mjs](../apps/osionos/app/scripts/bridge-api.mjs).

For local development, the root Compose stack runs `local-runtime-secrets`, which calls [infrastructure/baas/scripts/ensure-osionos-runtime-secrets.mjs](../infrastructure/baas/scripts/ensure-osionos-runtime-secrets.mjs). That script creates missing random local values in `.env.local` and configures osionos to require a bridge session by default. Use `--rotate` with the script when you intentionally want to invalidate local bridge sessions.

```bash
node infrastructure/baas/scripts/ensure-osionos-runtime-secrets.mjs
node infrastructure/baas/scripts/ensure-osionos-runtime-secrets.mjs --rotate
```

The effective runtime values are:

```bash
OSIONOS_BRIDGE_SHARED_SECRET=<generated locally>
OSIONOS_APP_SESSION_SECRET=<generated locally>
OSIONOS_BRIDGE_EMAIL_HASH_SALT=<generated locally>
OSIONOS_BRIDGE_URL=http://localhost:4000/api/auth/bridge/session
OSIONOS_APP_URL=http://localhost:3001
OSIONOS_ALLOWED_ORIGIN=http://localhost:3001
VITE_REQUIRE_BRIDGE_SESSION=true
VITE_PRISMATICA_URL=https://localhost:4323
```

Do not prefix those secrets with `VITE_` or `PUBLIC_`. Browser-visible variables are limited to URLs such as `VITE_API_URL` and `PUBLIC_OSIONOS_APP_URL`.

When `VITE_REQUIRE_BRIDGE_SESSION=true`, opening osionos directly shows a private-workspace gate instead of loading local seed data. Offline/demo mode is still possible only by explicitly setting `VITE_ALLOW_OFFLINE_MODE=true` for development.

## Tests

The osionos bridge tests live in [apps/osionos/app/tests/bridge/bridge-api.test.mjs](../apps/osionos/app/tests/bridge/bridge-api.test.mjs). They cover:

- valid signed handoff returns a redirect and owner workspace session
- tampered signatures are rejected
- timestamp replay windows are rejected
- BaaS persistence receives an email hash, not raw email or secrets
- missing app-session secrets stop before any BaaS persistence call
- bridge tokens are single-use

Run them with:

```bash
cd apps/osionos/app
pnpm test:bridge
```

The opposite-osiris login flow now calls the bridge after successful authentication and redirects to osionos only when the bridge returns a valid handoff URL. If osionos is unreachable, the website reports the bridge failure instead of silently pretending the app session exists.
