ADR-001: Monorepo Structureâ€‹

Status: Accepted
Date: 2026-02-18

Contextâ€‹

We need to organize code for a backend (NestJS), frontend (React), and shared types/utilities in a way that supports 4-5 developers working in parallel.

Decisionâ€‹

Use a monorepo structure with apps/ for applications and packages/ for shared code:

apps/backend/      # NestJS
apps/frontend/     # React + Vite
packages/shared/   # Shared types, DTOs, utilities

Rationaleâ€‹

  • Shared types: DTOs and interfaces defined once, used in both frontend and backend â€” eliminates type drift
  • Atomic PRs: A feature touching both frontend and backend lives in one PR
  • Simpler CI: One repo, one pipeline, one source of truth
  • No publishing overhead: Unlike a multi-repo setup, no need for private npm packages

Alternatives Consideredâ€‹

  • Separate repos (backend + frontend): rejected â€” too much overhead for a 4-person team, type sharing becomes painful
  • Nx/Turborepo: rejected â€” adds complexity we don't need for this project size

ADR-002: TypeScript Everywhere (Strict Mode)â€‹

Status: Accepted
Date: 2026-02-18

Contextâ€‹

The project involves multiple developers with varying experience levels working on both frontend and backend.

Decisionâ€‹

Use TypeScript in strict mode (strict: true) for all code â€” backend, frontend, and shared packages.

Rationaleâ€‹

  • Catches bugs at compile time â€” before they reach runtime or production
  • Self-documenting code â€” types serve as inline documentation
  • Refactoring confidence â€” rename a field and the compiler shows every place that needs updating
  • Shared language â€” everyone writes the same language, reducing context switching
  • Industry standard â€” TypeScript is used by most modern web projects; learning it is an investment

Configurationâ€‹

{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitReturns": true,
    "noFallthroughCasesInSwitch": true,
    "forceConsistentCasingInFileNames": true,
    "esModuleInterop": true,
    "skipLibCheck": true
  }
}

ADR-003: NestJS for Backendâ€‹

Status: Accepted
Date: 2026-02-18

Contextâ€‹

We need a backend framework that supports:

  • REST API endpoints
  • WebSocket connections (real-time features)
  • Authentication (JWT + OAuth)
  • Database integration (Prisma)
  • Modular architecture (multiple developers working in parallel)

Decisionâ€‹

Use NestJS 11 as the backend framework.

Rationaleâ€‹

  • Opinionated architecture â€” modules, controllers, services, guards, pipes, interceptors. Everyone follows the same patterns.
  • Built-in WebSocket support â€” @nestjs/websockets + Socket.IO or native WS
  • Passport.js integration â€” @nestjs/passport for JWT and OAuth strategies
  • Prisma integration â€” first-class recipe in NestJS docs
  • Swagger generation â€” automatic API documentation with @nestjs/swagger
  • Dependency injection â€” makes testing easier (mock services, swap implementations)
  • TypeScript native â€” built in TypeScript, for TypeScript

Alternatives Consideredâ€‹

  • Express.js: rejected â€” too unopinionated for a team project; everyone would structure code differently
  • Fastify: rejected â€” excellent performance but smaller ecosystem for auth/websockets
  • Django/Flask: rejected â€” team wants full-stack TypeScript

ADR-004: React + Vite for Frontendâ€‹

Status: Accepted
Date: 2026-02-18

Contextâ€‹

We need a frontend framework for a single-page application with real-time updates and a component-based architecture.

Decisionâ€‹

Use React 19 with Vite 7 as the build tool, Tailwind CSS v4 for styling, and shadcn/ui for accessible components.

Rationaleâ€‹

  • React 19: largest ecosystem, most learning resources, hooks + concurrent features
  • Vite 7: instant HMR, fast builds, native ESM â€” dramatically faster than Webpack
  • Tailwind CSS v4: utility-first, no CSS files to organize, consistent design system
  • shadcn/ui: accessible, customizable components (not a dependency â€” code is copied into your project)

Alternatives Consideredâ€‹

  • Next.js: rejected â€” SSR adds complexity we don't need for this SPA
  • Angular: rejected â€” steeper learning curve, heavier framework
  • Vue.js: rejected â€” team has more React experience

ADR-005: PostgreSQL as Primary Databaseâ€‹

Status: Accepted
Date: 2026-02-18

Contextâ€‹

We need a relational database for user accounts, game history, chat messages, and other structured data.

Decisionâ€‹

Use PostgreSQL 16 as the primary (and only) database.

Rationaleâ€‹

  • ACID compliance â€” data integrity for user accounts, scores, transactions
  • Rich feature set â€” JSONB, arrays, enums, full-text search, triggers, views
  • Row-Level Security â€” built-in RLS for fine-grained access control
  • Prisma support â€” first-class integration, excellent migration tooling
  • Industry standard â€” most deployed open-source relational database

ADR-006: Prisma as ORMâ€‹

Status: Accepted
Date: 2026-02-18

Decisionâ€‹

Use Prisma 7 as the database ORM.

Rationaleâ€‹

  • Type-safe queries â€” generated client provides full TypeScript autocompletion
  • Declarative schema â€” schema.prisma is readable and serves as documentation
  • Migrations â€” prisma migrate handles schema evolution safely
  • Prisma Studio â€” visual database browser for debugging (port 5555)
  • NestJS recipe â€” official integration guide in NestJS docs

ADR-007: Redis for Cache & Real-Timeâ€‹

Status: Accepted
Date: 2026-02-18

Decisionâ€‹

Use Redis 7 for caching, session storage, and real-time pub/sub.

Rationaleâ€‹

  • Pub/Sub â€” enables WebSocket message broadcasting across multiple server instances
  • Caching â€” reduces database load for frequently accessed data
  • Session store â€” faster than database-backed sessions
  • Rate limiting â€” backend for express-rate-limit or custom implementation
  • Simple â€” single dependency, Alpine image, no configuration needed

ADR-008: Docker-First Developmentâ€‹

Status: Accepted
Date: 2026-02-18

Decisionâ€‹

Use Docker Compose as the default development environment. Every team member runs make and gets an identical setup.

Rationaleâ€‹

  • Zero local dependencies â€” only Docker needed on the host
  • Identical environments â€” eliminates "works on my machine" issues
  • Database included â€” PostgreSQL + Redis start automatically
  • Named volumes â€” node_modules in Docker volumes (fast, isolated)
  • Production parity â€” dev environment mirrors production topology

ADR-009: JWT Authentication + OAuth 2.0â€‹

Status: Accepted
Date: 2026-02-18

Decisionâ€‹

Use JWT tokens (access + refresh) for API authentication, with OAuth 2.0 for third-party login (42 API / Google).

Rationaleâ€‹

  • Stateless â€” JWT tokens don't require server-side session storage
  • Refresh rotation â€” short-lived access tokens + rotating refresh tokens for security
  • OAuth 2.0 â€” industry standard for third-party authentication
  • Passport.js â€” battle-tested middleware with strategies for JWT and OAuth

ADR-010: Git Flow Branching Strategyâ€‹

Status: Accepted
Date: 2026-02-18

Decisionâ€‹

Use Git Flow with main (stable) and develop (integration) as protected branches. All work goes through feature branches and pull requests.

Rationaleâ€‹

  • Protected main â€” main always reflects production-ready code
  • Integration branch â€” develop is where features meet and get tested together
  • Code reviews â€” PRs enforce review before merge, catching bugs and sharing knowledge
  • Traceability â€” every change is linked to an issue and a PR
